# 1차 업무 — Git 활용 및 픽셀 단위 이미지 처리

Hugging Face `ethz/food101`에서 무작위 5장을 받아, 각 장마다 전처리 10단계를 모두 적용하고
이상치 여부를 판정한다. 모든 작업은 브랜치 → PR → 리뷰 → merge 흐름으로 관리했다.

## 실행

```bash
pip install -r requirements.txt

python image_preprocessing.py --num-samples 5 --seed 42   # 전처리 + 결과 이미지 생성
python pixel_ops.py                                       # NumPy 직접 구현 vs cv2 검증
```

## 파일

| 파일 | 역할 |
|---|---|
| `image_preprocessing.py` | 데이터 적재, 기본 전처리, 증강, 파이프라인 진입점 |
| `color_filter.py` | 특정 색상 픽셀 감지 및 필터링 (HSV) |
| `outlier.py` | 이상치 탐지 (어두운 이미지 / 작은 객체) |
| `pixel_ops.py` | 동일 연산의 NumPy 직접 구현 + cv2 대조 검증 |
| `preprocessed_samples/` | 처리 결과 5장 (각 장에 10단계 전부 포함) |

## 전처리 단계

각 이미지는 아래 10단계를 거치고, 결과는 `preprocessed_samples/{n}_{label}_steps.png`에
한 장으로 합쳐 저장된다.

| # | 단계 | 구현 | 왜 |
|---|---|---|---|
| 1 | 크기 조정 224×224 | `cv2.resize` (INTER_AREA) | 모델 입력 크기 통일. 축소에는 영역 평균이 가장 깔끔하다 |
| 2 | 그레이스케일 | `cv2.cvtColor(BGR2GRAY)` | 색상 정보를 뺀 형태 정보만 필요한 경우용 |
| 3 | 정규화 | ImageNet mean/std | 사전학습 모델의 입력 분포와 맞춤 |
| 4 | 노이즈 제거 | `cv2.GaussianBlur` 5×5 | 센서 노이즈·JPEG 아티팩트 완화 |
| 5 | 좌우 반전 | `cv2.flip` | 음식은 좌우가 뒤집혀도 같은 음식이라 안전한 증강 |
| 6 | 회전 +15° | `getRotationMatrix2D` + `warpAffine` | 촬영 각도 다양성 확보. 반사 패딩으로 검은 모서리 방지 |
| 7 | 색상 변화 | HSV 색조/채도/명도 조정 | 조명·카메라 화이트밸런스 차이를 흉내 |
| 8 | 색상 감지 | `cv2.inRange` (HSV) | 요청내용 2 — 특정 색상 픽셀 감지 |
| 9 | 색상 필터링 | `cv2.bitwise_and` | 감지된 픽셀만 남김 |
| 10 | 객체 마스크 | Otsu + `connectedComponentsWithStats` | 심화 문제 — 객체 면적 산출 |

### 왜 RGB가 아니라 HSV에서 색을 고르는가

RGB에서 "빨강"을 고르려면 밝기에 따라 임계값이 계속 흔들린다. HSV는 색상(H)이 밝기(V)와
분리돼 있어 조명이 달라져도 같은 H 범위로 잡힌다. 단 두 가지를 처리해야 한다.

- **빨강은 H=0에서 갈라진다.** OpenCV의 H는 0~179 순환이라 빨강이 양 끝(0~10, 170~179)에
  나뉘어 있다. 두 구간을 각각 `inRange`로 잡아 OR로 합친다.
- **무채색을 걸러야 한다.** 회색 접시는 `HSV=(0,0,128)`로 **H가 빨강과 똑같은 0**이다.
  채도(S) 하한을 두지 않으면 접시가 통째로 빨강으로 잡힌다. 합성 이미지로 확인한 결과
  S 하한 90을 두면 회색 배경 누출이 0픽셀이 된다.

## 이상치 탐지 (심화 문제)

| 기준 | 임계값 | 방법 |
|---|---|---|
| 너무 어두운 이미지 | 평균 밝기 < 40 | 그레이스케일 평균 |
| 객체가 너무 작은 이미지 | 최대 객체 면적 < 10% | Otsu 이진화 → 최대 연결요소 면적 / 전체 |

### 전경을 고르는 문제

Otsu는 밝기로 두 덩어리를 나눠줄 뿐 **어느 쪽이 피사체인지 모른다.** 음식 사진은 접시가
밝은 경우와 어두운 경우가 섞여 있어 "밝은 쪽이 음식"으로 고정할 수 없다. 그래서 마스크와
반전을 모두 후보로 두고 점수가 높은 쪽을 택한다.

```
전경다움 점수 = 중심 절반 영역 커버리지 − 테두리 점유율
```

두 단서를 **합쳐야** 하는 이유가 있다. 처음엔 테두리 점유만으로 골랐는데, `beignets`처럼
음식이 화면을 꽉 채운 사진에서 피사체가 테두리를 많이 건드려 배경으로 오인됐다. 그 결과
음식 대신 우측 하단 배경 조각을 잡고 면적을 15.7%로 보고했다 (PASS가 나오긴 했지만
우연이었다). 반대로 중심 커버리지만 쓰면, 작은 피사체를 감싼 배경이 중앙까지 덮고 있어
배경이 이긴다. 빼서 합치면 두 경우 모두 걸러지고, `beignets`는 66.7%로 정정된다.

### 이 지표의 한계

Otsu는 밝기로만 나누므로 이건 **진짜 객체 탐지가 아니라 면적 프록시**다. `tacos`에서는
밝은 접시·밥은 잡고 어두운 콩은 빠진다. "객체가 지나치게 작은 사진을 걸러낸다"는 목적에는
충분하지만, 정확한 객체 경계가 필요하면 3차 업무의 검출 모델이 맞다.

## 결과

`--seed 42` 기준 5장 모두 통과했다.

| # | 라벨 | 평균 밝기 | 객체 면적 | 판정 |
|---|---|---|---|---|
| 1 | peking_duck | 110.0 | 48.6% | PASS |
| 2 | tiramisu | 127.5 | 30.3% | PASS |
| 3 | beignets | 98.6 | 66.7% | PASS |
| 4 | ravioli | 139.1 | 42.5% | PASS |
| 5 | tacos | 121.4 | 51.8% | PASS |

무작위 5장이 전부 정상이라 제거 로직이 실제로 도는 걸 보여주지 못한다. 그래서 합성
이미지로 각 분기를 따로 검증했다.

| 케이스 | 평균 밝기 | 객체 면적 | 기대 | 판정 |
|---|---|---|---|---|
| 균일한 어두운 이미지 | 15.0 | 100% | 어두움으로 제거 | 어두움(평균 15.0) |
| 밝은 배경 위 20×20 객체 | 218.5 | 0.8% | 0.8% (=20²/224²) | 객체 작음(0.8%) |
| 밝은 배경 위 140×140 객체 | 145.8 | 39.0% | 39.1% (=140²/224²) | PASS |
| 어두운 배경 위 밝은 객체 | 102.5 | 39.0% | 극성이 반대여도 객체 선택 | PASS |

면적이 이론값과 일치하고, 네 번째 케이스에서 밝기 극성이 뒤집혀도 배경이 아닌 객체를
전경으로 고른다.

첫 케이스의 면적 100%는 퇴화 사례다. 값이 전부 같은 이미지는 Otsu가 나눌 경계가 없어
프레임 전체가 한 덩어리로 잡힌다. 밝기 검사가 먼저 걸러내므로 판정 자체는 옳지만,
"객체가 프레임을 100% 채운다"는 수치 자체에 의미는 없다.

## NumPy 직접 구현 대조 (`pixel_ops.py`)

cv2 함수가 픽셀 수준에서 무엇을 하는지 확인하려고 같은 연산을 NumPy로 다시 구현하고
결과를 비교했다. 최악 조건(균일 난수 이미지)에서도 네 연산 모두 일치한다.

| 연산 | 평균 절대오차 | 최대 절대오차 |
|---|---|---|
| grayscale | 0.0011 | 1.0 |
| gaussian blur 5×5 | 0.0021 | 1.0 |
| flip horizontal | 0.0000 | 0.0 |
| rotate 15° | 0.0004 | 1.0 |

**회전 수치는 cv2 버전에 딸린다.** 위 표는 cv2 5.0.0 기준이고, 4.12.0에서 같은 코드를
돌리면 회전만 평균 0.8141 / 최대 6.0이 나온다. 4.x의 `warpAffine`은 이중선형 보간
가중치를 5비트 고정소수점(1/32 단위)으로 양자화하는데 5.0은 더 높은 정밀도를 쓴다.
최대 오차 6/255가 딱 그 양자화 폭과 맞아떨어진다. 난수 이미지는 이웃 픽셀 사이에
상관이 없어 좌표가 조금만 틀어져도 오차가 커지는 최악 조건이라 차이가 도드라진다.
실제 사진에서는 이 차이가 눈에 띄지 않고, 전처리 결과와 이상치 판정은 두 버전이 같다.

여기까지 오는 데 두 가지를 고쳐야 했고, 둘 다 대조를 안 했으면 못 찾았을 것들이다.

**1. cv2는 작은 커널에서 가우시안 공식을 안 쓴다.** `sigma<=0`이고 `ksize<=7`이면 공식
대신 하드코딩된 이항계수 테이블(`smallGaussianTab`, 5×5는 `[1,4,6,4,1]/16`)을 쓴다.
교과서 공식대로 구현하니 평균 오차 0.73이 남았다. 같은 분기를 재현해서 0.0021로 떨어졌다.

**2. `getRotationMatrix2D`는 순방향 행렬을 준다.** `warpAffine`이 내부에서 뒤집어 쓴다.
역방향 매핑(출력 픽셀에서 원본 좌표를 거꾸로 찾는 방식)을 직접 짤 때 이 행렬을 그대로
쓰면 **반대 방향으로 회전한다.** 오차 57.8로 잡혔고, 역행렬로 바꾸니 0.0004가 됐다.

## Git 워크플로우

브랜치를 기능 단위로 끊고 PR로 합쳤다.

```
main
 ├── feature/image-processing   기본 전처리 + NumPy 대조 검증
 ├── feature/color-detection    HSV 색상 감지/필터링
 ├── feature/outlier-filter     이상치 탐지
 └── feature/pipeline-report    파이프라인 + 결과 이미지 + 문서
```

사용한 명령은 아래와 같다.

```bash
git clone https://github.com/park-hyun-su/comento.git
git checkout -b feature/image-processing
git add . && git commit -m "feat: ..."
git push -u origin feature/image-processing
gh pr create --base main --head feature/image-processing
# 리뷰 후
git checkout main && git merge feature/image-processing && git push origin main
```
